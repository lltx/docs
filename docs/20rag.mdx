# [基于上下文的检索：增强 AI 模型知识检索能力](https://www.anthropic.com/news/contextual-retrieval)



<img src='https://minio.pigx.top/oss/202410/1728399617.png' alt='1728399617'/>

对于AI模型在特定上下文中使用，通常需要访问背景知识。例如，客户支持聊天机器人需要了解特定业务的相关知识，而法律分析师机器人则需要了解大量过往案例。开发者通常使用**检索增强生成**（RAG）来增强AI模型的知识。RAG是一种从知识库中检索相关信息并将其附加到用户提示中的方法，显著增强了模型的响应。然而，传统的RAG解决方案在编码信息时会丢失上下文，导致系统无法从知识库中检索到相关信息。

在本文中，我们将介绍一种显著提高RAG检索步骤性能的方法：**上下文检索**。该方法使用两种子技术：**上下文嵌入**（Contextual Embeddings）和**上下文BM25**，可以将检索失败率降低49%，结合重排序技术甚至可以降低67%，显著提升系统性能。

你可以使用Claude轻松部署自己的上下文检索解决方案，参考[我们的用法](https://github.com/anthropics/anthropic-cookbook/tree/main/skills/contextual-embeddings)。

## 使用RAG的场景

有时候最简单的方法是最好的。如果你的知识库小于200,000个标记（大约500页的材料），你可以将整个知识库包含在提供给模型的提示中，而不需要RAG或类似的方法。

几周前，我们为Claude发布了[提示缓存](https://docs.anthropic.com/en/docs/build-with-claude/prompt-caching)，使得这种方法显著更快并且更节省成本。开发者现在可以在API调用之间缓存经常使用的提示，将延迟减少超过2倍，并将成本降低多达90%。[提示缓存食谱](https://github.com/anthropics/anthropic-cookbook/blob/main/misc/prompt_caching.ipynb)详细讲解了它的工作原理。

然而，随着你的知识库的增长，你将需要一个更可扩展的解决方案，这就是**上下文检索**的用武之地。

## RAG：扩展到更大的知识库

对于那些不适合上下文窗口的知识库，RAG是典型的解决方案。RAG通过以下步骤预处理知识库：

1. 将知识库（文档的“语料库”）分解为较小的文本块，通常不超过几百个标记；
2. 使用嵌入模型将这些块转换为向量嵌入，这些嵌入编码意义；
3. 将这些嵌入存储在允许通过语义相似性搜索的向量数据库中。

在运行时，当用户向模型输入查询时，使用向量数据库根据语义相似性找到最相关的块。然后，将最相关的块添加到发送给生成模型的提示中。

虽然嵌入模型在捕捉语义关系方面表现出色，但它们可能会错过关键的精确匹配。幸运的是，BM25（最佳匹配25）是一种可以解决这个问题的传统技术。它通过词汇匹配的排名函数，用于找到精确的单词或短语匹配。BM25尤其适用于包含唯一标识符或技术术语的查询。

### BM25与嵌入模型结合

RAG解决方案可以通过以下步骤结合嵌入和BM25技术来更准确地检索最相关的块：

1. 将知识库分解为较小的文本块，通常不超过几百个标记；
2. 为这些块创建TF-IDF编码和语义嵌入；
3. 使用BM25根据确切匹配找到最相关的块；
4. 使用嵌入根据语义相似性找到最相关的块；
5. 使用**Rank Fusion**技术结合并去重BM25和嵌入的结果；
6. 将最相关的块添加到提示中以生成响应。

通过结合BM25和嵌入模型，传统的RAG系统可以提供更全面和准确的结果，平衡精确的词匹配与语义理解。

然而，传统RAG系统在上下文处理方面存在局限性，文档通常被拆分为较小的块以进行高效检索。当单个块缺乏足够的上下文时，检索到的信息可能不够准确。

## 上下文检索

**上下文检索**通过在每个块之前添加特定于块的解释性上下文，在嵌入和BM25索引之前解决了上述问题。

举个例子，假设你正在处理美国SEC文件的财务数据集，并收到以下问题：

> "ACME公司在2023年第二季度的收入增长是多少？"

相关的块可能包含以下文本：

> "公司收入比上一季度增长了3%."

然而，这个块本身没有指明它指的是哪家公司或相关时间范围。通过添加上下文，我们可以将该块转换为如下形式：

```plaintext
original_chunk = "该公司的收入比上一季度增长了3%。"

contextualized_chunk = "该段落摘自ACME公司2023年第二季度的SEC文件；上一季度的收入为3.14亿美元。该公司的收入比上一季度增长了3%。"
```

### 实现上下文检索

为了自动化这个过程，我们使用Claude生成上下文。以下是Claude的提示示例，用于为每个块生成上下文：

```plaintext
<document> 
{{WHOLE_DOCUMENT}} 
</document> 
这是我们希望在整个文档中定位的段落。
<chunk> 
{{CHUNK_CONTENT}} 
</chunk> 
请提供一个简短的上下文，以将该段落在整个文档中定位，以便改善对该段落的搜索检索。只需给出简短的上下文，不要包含其他内容。
```

生成的上下文文本通常为50-100个标记，附加到每个块之前，进行嵌入和BM25索引。

## 实现考虑

在实现上下文检索时，有几点需要考虑：

1. **块的边界**：如何将文档拆分为块会影响检索性能。块大小、边界选择以及重叠部分的设计尤为重要。
2. **嵌入模型**：上下文检索在所有测试的嵌入模型中都提高了性能，特定的模型可能效果更好。我们发现Gemini和Voyage嵌入效果尤为突出。
3. **自定义提示**：虽然提供的通用提示效果不错，但为特定领域定制的提示可能会获得更好的结果。例如，在提示中引入一个词汇表来解释知识库中的特定术语。
4. **块的数量**：更多的块有助于提高相关性，但过多的块可能导致分散注意力。我们发现，使用20个块的效果最好。

## 重排序

在传统的RAG系统中，初始检索通常会返回大量块，具有不同的相关性和重要性。通过使用**重排序**技术，可以确保只将最相关的块传递给模型，从而提升性能并减少处理开销。

### 重排序的步骤：

1. 执行初始检索，获取前150个可能相关的块；
2. 使用重排序模型对这些块进行重新排序，以确保最高质量的内容用于最终生成。
